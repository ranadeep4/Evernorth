POM & Custom Fixtures

Building Maintainable & Scalable Test Code

Page Object Model

  • Locators centralized in page classes
  • Reusable actions across all tests
  • Update once, fix everywhere
  • Clean separation of concerns
  • Private fields for encapsulation

Custom Fixtures

  • Dependency injection pattern
  • Auto page object creation via base.extend
  • Zero boilerplate in test files
  • Full test isolation & parallel safety
  • Lazy-loaded page instances
export const test = base.extend<{ loginPage: LoginPage; homePage: HomePage }>({
  loginPage: async ({ page }, use) => await use(new LoginPage(page)),
  homePage: async ({ page }, use) => await use(new HomePage(page))
});

Best Practices I Learned

Industry Standards & Guidelines

Code Quality

  • Meaningful test names that describe intent
  • Keep tests independent — no shared state
  • Follow DRY — reuse page objects
  • Avoid hardcoded waits (use auto-wait)

Project Structure

  • Separate pages/, tests/, data/ folders
  • External test data (JSON/TS)
  • Configuration files (playwright.config)
  • Version control everything

Performance

  • Parallel execution with workers
  • Use API for test setup when possible
  • Minimize unnecessary UI interactions
  • Reuse browser contexts

AI-Powered Failure Analysis

The Highlight Feature

How It Works

  • Custom Playwright Reporter (AIAnalyzerReporter)
  • Hooks into onTestEnd lifecycle event
  • Captures failed test name, error, stack trace
  • Sends to Google Gemini 2.5 Flash API
  • Saves JSON report with AI suggestions

AI Response Includes

  • Category — type of failure detected
  • Root Cause — why the test failed
  • Suggested Fix — actionable steps
  • Confidence — score (0 to 1)
  • All saved to ai-analysis-report.json

AI Analysis — Real Example

Actual AI Output from Test Failures

Album Filter Failure

  • Error: Expected 0 items, found 18
  • Root Cause: Filter didn't apply — UI still showed all items instead of empty set
  • AI Fix: Verify filter input is accepted, add assertion to confirm filter text, debug filtering logic
  • Confidence: 0.95

Play Song Failure

  • Error: Expected truthy, received false
  • Root Cause: Song playback didn't start — play action did not produce expected state
  • AI Fix: Verify playback logic, check audio dependencies, handle async properly with await
  • Confidence: 0.95

Impact: AI suggestions reduced debugging time significantly by pointing directly to the root cause

API Testing — Approach

How I Built & Organized API Tests

Testing Strategy

  • 6 test files — one per HTTP method + negative
  • Fixture-based test data (mobileDevices.json)
  • Create-then-test pattern for GET, PUT, PATCH, DELETE
  • Status code validation (200, 201, 204)
  • Response body property & type assertions
  • Error handling with failOnStatusCode: false

Fixture Data

  • 5 mobile devices (iPhone, Samsung, Pixel, iPad, MacBook)
  • Each with price, color, capacity, CPU, screen size
  • Negative payloads — missing name field
  • Partial update payload for PATCH tests
  • Data loaded via cy.fixture() in before() hooks
  • Dynamic object creation before each test

Result: Complete CRUD coverage with positive and negative scenarios, fixture-driven data, and proper assertions
