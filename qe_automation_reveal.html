JavaScript Fundamentals (Revision)

  • Variables: let, const, avoid var
  • Types: primitives vs objects
  • Equality: prefer === over ==
  • Arrays: map/filter/reduce

"use strict";
const fruits = ["Apple", "Banana", "Cherry"];
const upper = fruits.map(f => f.toUpperCase());

// Equality: avoid coercion surprises
"5" == 5;   // true
"5" === 5;  // false
Remind: tests fail often due to coercion + shared mutable arrays.

Async Patterns You Must Know

  • Callbacks → Promises → async/await
  • Rule: await the thing you depend on
  • Parallel: use Promise.all when safe
  • Don’t mix implicit waits with manual sleeps

// Sequential (clear)
await page.goto(url);
await page.getByRole('button', { name: 'Login' }).click();

// Parallel (fast) when independent
await Promise.all([
  page.waitForURL(/dashboard/),
  page.getByRole('button', { name: 'Login' }).click(),
]);
Explain why Promise.all reduces flaky timing issues when used correctly.

Element Locator Strategies

  • Prefer getByRole / accessible locators
  • Use data-testid when needed
  • Avoid brittle CSS chains & XPath
  • Assert on visible user outcomes

await page.getByRole('textbox', { name: 'Username' }).fill('Admin');
await page.getByRole('textbox', { name: 'Password' }).fill('admin123');
await page.getByRole('button', { name: /login/i }).click();

// data-testid fallback
await page.getByTestId('submit').click();
Stability > cleverness. Make locators read like requirements.

Playwright Fundamentals

  • Auto-waits for actions + assertions
  • Parallel execution (workers) by default
  • Tracing + screenshots + video help debugging
  • Golden rule: assert what the user sees

From the revision guide: Playwright supports multi-browser and modern async/await flows.


import { test, expect } from '@playwright/test';

test('valid login', async ({ page }) => {
  await page.goto('https://.../auth/login');
  await page.getByPlaceholder('Username').fill('Admin');
  await page.getByPlaceholder('Password').fill('admin123');
  await page.getByRole('button', { name: 'Login' }).click();
  await expect(page.getByRole('heading', { name: 'Dashboard' })).toBeVisible();
});
Mention auto-wait + locators reduce flakiness.

Cypress Fundamentals

  • Command queue + implicit retries
  • Great dev feedback loop
  • Strong network stubbing (cy.intercept)
  • Fixtures: JSON test data

cy.visit('/login');
cy.get('#user').type('demo');
cy.get('#pass').type('demo');
cy.get('button[type=submit]').click();

cy.intercept('GET', '/api/**').as('api');
cy.wait('@api');
Explain difference: Cypress chains vs Playwright await.

Page Object Model (POM)

  • Encapsulate locators + actions
  • Tests read like business flows
  • One change updates many tests
  • Keep page objects action-focused (not assertion-heavy)

export class LoginPage {
  constructor(private page: Page) {}
  async login(user: string, pass: string) {
    await this.page.getByPlaceholder('Username').fill(user);
    await this.page.getByPlaceholder('Password').fill(pass);
    await this.page.getByRole('button', { name: 'Login' }).click();
  }
}
Highlight: POM reduces duplication and maintenance cost.

Custom Fixtures & Base Test Extension

  • test.extend() injects POMs as fixtures
  • No repetitive beforeEach boilerplate
  • Each test declares what it needs
  • Cleaner parallel execution

import { test as base } from '@playwright/test';

export const test = base.extend<{ loginPage: LoginPage }>({
  loginPage: async ({ page }, use) => {
    await use(new LoginPage(page));
  }
});
Connect to your framework: tests only import basetest + data.

Data‑Driven Testing

  • Run the same test with multiple datasets
  • Sources: JSON, Excel, DB (as per guide)
  • Keep test cases independent for parallel runs
  • Make failures readable: include case description

const data = [
  { username: 'user1', password: 'pass1', expected: 'Dashboard' },
  { username: 'user2', password: 'bad', expected: 'Error' },
];

data.forEach(tc => {
  test(`login: ${tc.username}`, async ({ page }) => {
    // ... steps + assertion
  });
});
Emphasize: data-driven reduces duplication and improves coverage.

API Testing (Playwright)

  • CRUD validation: POST/GET/PUT/PATCH/DELETE
  • Token auth (cookie/header)
  • Always dispose request context

Guide examples use `request.newContext({ baseURL, headers })`.


const api = await request.newContext({
  baseURL: 'https://jsonplaceholder.typicode.com',
  extraHTTPHeaders: { 'Content-Type': 'application/json' },
});

const res = await api.post('/posts', {
  data: { title: 'foo', body: 'bar', userId: 1 }
});
expect(res.status()).toBe(201);
await api.dispose();
Mention: API tests are faster and reduce UI dependence.

Cypress vs Playwright (Quick Compare)

Cypress

  • Runs inside browser (command queue)
  • Great DX + stubbing
  • Multi-tab limitations
  • Best for frontend-heavy apps

Playwright

  • Controls browser externally
  • Native multi-tab + multi-origin
  • Strong parallel + multi-browser
  • Great for end-to-end + CI scale
Conclude: choose based on product + team + CI needs.
